# KBO 데이터 대시보드 — Steering Document

> 이 문서는 프로젝트의 "헌법"입니다.
> 모든 설계, 구현, 의사결정은 이 문서의 원칙을 따릅니다.
> AI 에이전트(Cursor, Claude Code)에게 이 문서를 항상 참조하게 합니다.
> 최종 수정일: 2026-03-16

---

## 1. 프로젝트 비전

### 한 줄 정의
**"네이버가 못 하는 KBO 복합 조건 질의를 드롭다운 조합으로 해결하는 데이터 대시보드"**
— STATIZ보다 깔끔하고, 네이버보다 깊게

### 왜 이것을 만드는가
- KBO 1,200만 관중 시대에 데이터에 관심 있는 팬이 늘고 있지만, 복합 조건 질의를 할 수 있는 서비스가 없다
- "2024년 좌타자 상대 FIP가 가장 낮은 투수 5명" 같은 질문은 네이버로 안 되고, STATIZ에서도 불편하다
- STATIZ가 2025년 개편 후 신뢰도가 하락하면서, 더 나은 데이터 서비스 진입 타이밍이 열렸다
- 이 프로젝트를 통해 SDD 프로세스 + 풀스택 개발 + 데이터 파이프라인을 실전에서 경험한다

### 성공의 정의
- MVP 완성 후 실제 KBO 데이터로 복합 조건 질의가 동작한다
- STATIZ/KBReport 유저가 "이거 편하다"고 느낀다
- 포트폴리오로서 풀스택 + 데이터 엔지니어링 역량을 증명한다

---

## 2. 핵심 원칙

### P1. 드롭다운이 핵심이다
- 이 서비스의 가치는 "드롭다운 5단 조합으로 복합 조건 질의"에 있다
- 모든 기능은 이 핵심을 보조하거나 유저를 이쪽으로 유도하는 역할이다
- AI/SLM은 보너스. 드롭다운 없이는 서비스가 안 되지만, AI 없이는 서비스가 돌아간다

### P2. 네이버와 경쟁하지 않는다
- 일정, 스코어, 기본 라인업은 네이버가 이미 완벽하게 한다
- 우리는 네이버가 못 하는 것만 한다: 복합 조건 질의, 세이버메트릭스, 상황별 스플릿
- 일정/결과 기능을 넣되, 그 자체가 목적이 아니라 탐색기로 가는 진입점으로만 활용한다

### P3. 데이터 정확도가 생명이다
- 세이버메트릭스 계산 하나 틀리면 신뢰가 무너진다 (STATIZ가 망한 이유)
- 다중 소스 크로스체크 (KBO 공식 + STATIZ + KBReport)를 반드시 수행한다
- 자체 계산 엔진의 결과를 기존 소스와 비교 검증하는 TDD를 적용한다

### P4. 하나를 제대로 만든다 (빌드 트랩 방지)
- "에이전트가 빠르니까 더 많이 넣자"는 유혹을 경계한다 (교재 Ch50)
- 1순위 MVP를 완성하기 전에 2순위 기능에 손대지 않는다
- 기능 추가는 반드시 우선순위표(1순위→2순위→3순위)를 따른다

### P5. 비용 0원을 유지한다
- LLM API 비용, 유료 호스팅, 유료 데이터 소스를 사용하지 않는다
- 모든 기술 선택은 "무료로 가능한가"를 먼저 확인한다
- 향후 수익이 발생하면 그때 유료 인프라로 전환한다

## 3. 제약 조건

### 변경 불가 (하드 제약)
| 제약 | 설명 |
|-----|------|
| 비용 0원 | LLM API, 유료 호스팅, 유료 데이터 모두 불가 |
| 1인 개발 | 모든 설계, 구현, 테스트, 배포를 혼자 한다 |
| GPU 한계 | GTX 1660 (6GB VRAM) — 대규모 ML 학습 불가 |
| KBO API 부재 | 공식 API 없음. 크롤링 + 기존 라이브러리로 데이터 확보 |

### 변경 가능 (소프트 제약)
| 제약 | 현재 선택 | 변경 조건 |
|-----|----------|----------|
| DB | SQLite | 데이터 규모 커지면 PostgreSQL로 전환 |
| 배포 | Render 무료 티어 | 트래픽 증가 시 유료 전환 또는 프론트/백 분리 |
| 데이터 범위 | 미정 | 데이터 수집 파이프라인 구축 후 결정 |
| 세이버메트릭스 범위 | 풀 (wOBA, xFIP, BABIP, DRS 등) | 데이터 소스 제약에 따라 조정 가능 |

## 5. 기술 선택과 근거

### 기술 스택

| 영역 | 선택 | 근거 |
|-----|------|------|
| 프론트엔드 | React + Chart.js | 대시보드 UI에 최적화. 컴포넌트 기반으로 15개 기능을 모듈화 가능 |
| 백엔드 | Python + FastAPI | Python은 데이터 처리(pandas) + ML(sklearn)에 강점. FastAPI는 비동기 + 자동 API 문서화 |
| DB | SQLite → PostgreSQL | MVP는 SQLite로 빠르게. 데이터 커지면 PostgreSQL로 무중단 전환 (SQLAlchemy ORM) |
| 배포 | Render 무료 티어 | FastAPI가 React 정적 파일도 서빙. 1인 개발이라 인프라 관리 포인트 최소화 |
| 데이터 수집 | kbo-data + 직접 크롤링 | 비용 0원. 기존 라이브러리로 빠르게 시작, 부족한 부분만 직접 크롤링 |
| 세이버 계산 | 자체 엔진 (Python) | FanGraphs 공식 + KBReport 리그 상수 기반. pybaseball 소스코드 참고 |
| AI (향후) | 로컬 SLM + Ollama | 비용 0원 유지. 실험적 — 안 되면 드롭다운으로 충분 |

### 사용하지 않는 것과 근거

| 기술 | 사용 안 함 | 근거 |
|-----|----------|------|
| Vercel | 프론트 배포 | FastAPI 백엔드와 궁합 안 맞음 |
| LLM API (OpenAI, Claude) | AI 질의 | 비용 0원 제약 |
| Next.js | 프론트엔드 | 서버 렌더링 불필요. React CSR로 충분 |
| MongoDB | DB | 관계형 질의가 핵심이라 SQL 기반이 맞음 |
| 좌석 시야 기능 | 기능 | 자리어때 저작권으로 불가 |

## 6. 기능 우선순위 (경계선)

### 이것은 한다 (In Scope)

**1순위 MVP — 반드시 완성:**
- ① 데이터 탐색기 (드롭다운 5단 + 결과 테이블 + 차트)
- ② 선수 세부정보 (클래식 + 세이버 + 스플릿 3탭)
- ③ 순위 (팀 순위 + 팀스탯 종합 비교)

**2순위 — MVP 직후:**
- ④ 일정/결과 (탐색기 진입점)
- ⑤ 풀 라인업 (선발 + 불펜 + 벤치)
- ⑥ 기록 조회 (줌아웃 테이블)

**3순위 — 유저 확보 후:**
- ⑦ 이색 통계, ⑧ 핫/콜드, ⑨ 커뮤니티, ⑩ 직관로그, ⑪ 응원가, ⑫ SNS 허브

### 이것은 안 한다 (Out of Scope)
- 좌석 시야/리뷰 (자리어때 저작권)
- 티켓 예매 연동 (B2B 필요, 1인 불가)
- 굿즈 중고거래 (별도 서비스 영역)
- 구장 F&B 주문 (B2B 필요, 1인 불가)
- 실시간 중계 (중계권/저작권)
- 네이버가 이미 잘 하는 것 (기본 스코어, 기본 일정)

### 실험적 (Maybe — 조건부)
- ⑬ AI 자연어 질의 → GTX 1660으로 가능한 경우에만. 안 되면 드롭
- ⑭ 승부 예측 → 데이터 축적 후
- ⑮ SNS 콘텐츠 자동 생성 → 유저 확보 후

## 7. 데이터 원칙

### 데이터 소스 우선순위
1. KBO 공식 홈페이지 (koreabaseball.com) — 가장 신뢰할 수 있는 클래식 기록
2. KBReport (kbreport.com) — 세이버메트릭스 + KBO 리그 상수
3. STATIZ (statiz.co.kr) — 보조 크로스체크용
4. Baseball-Reference / FanGraphs — 해외 소스 보조
5. kbo-data 라이브러리 — 벌크 수집 도구

### 데이터 수집 원칙
- 한번 확보한 데이터는 자체 DB에 저장하여 외부 의존도 최소화
- 시즌 중 주기적 업데이트 (매일 또는 매주 배치)
- 크롤링은 예의 바르게 (rate limiting, robots.txt 준수)
- 데이터 정합성 검증 자동화 (TDD로 계산 결과 테스트)

### 세이버메트릭스 계산 원칙
- 공식 출처: FanGraphs Library + KBReport STAT Dic
- KBO 리그 상수는 MLB와 다르므로 반드시 KBO용으로 조정
- 모든 계산 함수에 단위 테스트 작성 (TDD — Ch46)
- 계산 결과를 KBReport/STATIZ와 크로스체크

## 8. 개발 원칙

### SDD 프로세스 준수
```
steering (이 문서) → requirements → design → tasks → implementation → 반복
```
- 코드 작성 전에 반드시 스펙을 먼저 정의한다
- AI 에이전트에게 지시할 때 반드시 스펙 문서를 @file로 참조시킨다
- "바로 만들어줘"는 금지. "이 스펙대로 만들어줘"가 원칙

### AI 에이전트 활용 원칙
- 에이전트는 도구이지 의사결정자가 아니다 (교재 Ch49)
- 4가지 역할로 분리: ①설계 보조자 ②비판적 리뷰어 ③코드 생성기 ④학습 도우미
- 에이전트가 생성한 코드는 반드시 Git diff로 확인한다
- 반복 CRUD는 Agentic으로 위임, 핵심 로직(동적 SQL, 세이버 계산)은 인간이 검증

### 코드 품질 원칙
- TDD: 세이버메트릭스 계산, API 엔드포인트에 테스트 먼저 작성 (Ch46)
- Git: 작은 단위로 커밋. 기능별 브랜치. PR로 변경 리뷰
- 보안: .env에 민감 정보, .gitignore 필수. API 키 절대 커밋 금지 (Ch43)
- 코드 스타일: Python은 PEP 8, React는 Cursor Rules에서 정의

### 배포 원칙
- MVP는 Render 무료 티어에 FastAPI + React 통합 배포
- 배포 전 로컬에서 전체 기능 테스트 완료
- 배포 후 모니터링 (에러 로그 확인)

## 10. 경쟁 환경 인식

### 우리의 빈틈 (차별화)
| 항목 | KBO 공식 | STATIZ | KBReport | 우리 |
|-----|---------|--------|----------|-----|
| 클래식 | O | O | O | O |
| 세이버메트릭스 | X | O (신뢰↓) | O | O (자체 계산) |
| 스플릿 | X | O | 일부 | O |
| **복합 조건 질의** | **X** | **제한적** | **X** | **핵심** |
| UX | 올드 | 복잡 | 깔끔 | 깔끔 + 그래프 |

### 솔직한 리스크 인식
- 타겟 시장이 니치 (KBO 팬 중 데이터 팬은 소수)
- 데이터 정확도 유지가 1인으로 힘들 수 있음
- KBReport가 이미 STATIZ 대안으로 자리잡는 중
- 수익 모델 불투명 → 포트폴리오/학습 가치로 보완

### 이 리스크를 줄이는 전략
- 이색 통계(⑦)와 핫/콜드(⑧)로 바이럴 유입 → 니치 시장 확장
- TDD로 데이터 정확도 자동 검증 → 1인 유지보수 부담 감소
- "데이터 + 커뮤니티" 결합으로 KBReport와 차별화
- 최악의 경우에도 포트폴리오로서의 가치는 확실

## 11. 의사결정 원칙

새로운 기능이나 기술을 추가할 때 이 체크리스트를 따른다:

**추가 전 질문:**
1. 이것이 핵심(드롭다운 탐색기)을 강화하는가?
2. 1순위 MVP가 완성된 후인가?
3. 비용 0원을 유지하는가?
4. 1인이 유지보수할 수 있는가?
5. 네이버가 이미 하고 있는 것은 아닌가?

→ 5개 모두 YES여야 추가한다.
→ 하나라도 NO면 보류하거나 우선순위를 낮춘다.
